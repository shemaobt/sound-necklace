# Noridoc: ui/app

Path: @/ui/app

### Overview

- The composition root and wiring layer of the Shemá app (ENG-224): the only place that imports adapters, @/domain, @/ui/state and @/ui/organisms together. It is the durable shell — station/adapter/addon issues almost always land purely by ADDING files under @/ui/pages, @/adapters or `ui/app/addons`. Rare, deliberate one-time exceptions wire a page that needs shell-held ports (ENG-270 made the Export tail reachable and injects the sessions store + sessionId — see Things to Know).
- Owns the app frame (@/ui/app/header.tsx brand + sound toggle), the "fio de contas" progress stepper, the single itinerant player (now with a live audio `Player` built per-session, ENG-275), review/lock chrome, the online-only connection gate, and the station host — all mounted from @/ui/app/App.tsx. It also wires the three resilience seams into the running session view (ENG-277): offline (window `online`/`offline` events), mid-session auth expiry (→ `/login`), and an advisory lock held by another editor (→ review chrome).
- App entry chain: @/index.html → @/ui/app/main.tsx (loads token side-effects once) → @/ui/app/App.tsx.

### How it fits into the larger codebase

- Outermost wiring layer of the atomic-design ladder documented in @/ui/docs.md. Per @/.dependency-cruiser.cjs it may import everything (adapters, domain, state, organisms); atoms/molecules/organisms below it may not reach back up.
- **Owns the three add-a-file registries** (@/docs/architecture.md §4), created once in @/ui/app/registries.ts and never edited again:

  | Registry | Glob                      | Builder                | Keyed by       |
  | -------- | ------------------------- | ---------------------- | -------------- |
  | Stations | `/ui/pages/*/index.tsx`   | `buildStationRegistry` | directory name |
  | Adapters | `/adapters/*/register.ts` | `buildAdapterRegistry` | port name      |
  | Addons   | `/ui/app/addons/*.tsx`    | `listAddons`           | sorted by path |

  This is the loop's no-file-conflict guarantee: landing a station, adapter, or overlay = adding files, never touching a shared registry. The pattern is proven: the first addon (@/ui/app/addons/tutorial.tsx, the tutorial popup, ENG-231) landed without editing a single shell file.

- Reads domain decisions through @/ui/state stores (never mutating @/domain directly): `useSessionStore`/`useAppStore` supply session state, the review/lock/online gate flags, and the sound toggle. Stepper navigation dispatches `setMode` through `sessionStore.apply` so the domain gates decide whether the move is allowed.
- Subscribes the `connectivity` adapter port (fixture default, @/adapters/connectivity) and mirrors online state both into a local flag for @/ui/organisms/connection-gate and into the session store, which pauses editing when offline.

```
index.html → main.tsx → App.tsx (composition root)
   ├─ Header (brand + sound toggle → appStore)
   ├─ Stepper (fio de contas, stepper-model ← domain modeLocks)
   ├─ ReviewBanner (review/lock chrome ← sessionStore)
   ├─ PlayerSlotProvider (the one traveling player)
   │    └─ ConnectionGate (online) → StationHost → ui/pages station
   └─ AddonsLayer (ui/app/addons overlay chrome)
```

### Core Implementation

- **Router** (@/ui/app/router.tsx): a minimal History-API router — react-router is out of scope. `matchRoute` resolves `/login`, `/dashboard` (also `/`), `/setup`, `/session/:id`, else `unknown`. `navigate()` calls `pushState`/`replaceState` and then notifies subscribers by hand, because pushState never fires `popstate`; `subscribe` additionally listens for `popstate` (browser back/forward). `usePathname`/`useRoute` read through `useSyncExternalStore` with `location.pathname` as the snapshot (a primitive, so `Object.is` comparison is stable).
- **The itinerant player** (@/ui/app/player-slot.tsx, PRD §8.0 / redesign §5.2): there is ONE player instance that travels to the active station's slot. Because changing a `createPortal` container remounts the subtree (react#12247), the player node lives in a persistent detached `holder` div (lazy `useState` init) that the portal always targets; the holder is MOVED between `PlayerHost` slots via `appendChild` (append moves, does not clone) in a `useLayoutEffect`. Placement runs through refs (no `setState` during commit), so a late-mounting lazy station host still receives the player. On active-station change it calls `player.stop()` — fed `player ?? NO_PLAYBACK` (a module-level no-op `{ stop() {} }`) so navigating between stations still stops audio, mirroring the reference `setMode`→`stopPlayback`.
- **The session audio player** (@/ui/app/audio-player.ts + @/ui/app/App.tsx `useSessionPlayer`, ENG-275, PRD §8.2): the composition root builds the actual audio `Player` for the active session and threads it to the necklace station (this is the audio analog of ENG-270's Export reachability). `buildSessionPlayer(sessionId)` re-decodes the session's bucket audio (`meta.bucketAudioId` from the persisted DTO via `appSessionStore().load`) through a @/adapters/audio `FixtureAudioEngine` + @/adapters/bucket `FixtureBucketSource`, builds the `Player` with the session's locked `beadSec`, and starts the clock bridge; it rejects if the session has no saved state or the bucket audio does not resolve. `startClockBridge(transport)` drives the `FixtureTransport` — whose clock only advances on `advance()` — from `requestAnimationFrame` using the real inter-frame delta, so playback progress + `onHead` (and thus the necklace `data-play` playhead) advance in the running browser (Playwright ticks rAF); it returns a canceller. `useSessionPlayer` rebuilds on session change and, on cleanup, stops the player + cancels the bridge; a session whose audio can't resolve degrades to a dormant `null` player (stations already handle `player = null`). It REPLACED the old no-op stub player.
- **The fio-de-contas stepper** (@/ui/app/stepper-model.ts + @/ui/app/stepper.tsx): `stepperStations` derives the six station states (Ouvir · Cortar · Triagem · Frases · Conversa · Guardar) from domain `modeLocks()` (@/domain) plus mode/flow position. `reachable` mirrors the gates; `state` is done/current/future by flow index. The Guardar (export) tail has no domain mode — the domain `Mode` is only escuta|triagem|segmentacao|mapeamento — so its `reachable` equals the Conversa/mapeamento gate (`modeLocks().mapeamento`: whole story confirmed + ≥1 locked phrase in a productive scene), and entering it is a shell-local concern: the optional `{ viewingExport }` option marks Guardar as `current` (earlier beads as `done`) without any domain mode change. It is a progress indicator, not free navigation — the `<ol>` uses one delegated click handler mapped by `<li>` index, and only navigates to `reachable` stations. Zero digits render (PRD §9.2); the station `key` doubles as the @/ui/pages directory name the station registry resolves.
- **Review/lock chrome** (@/ui/app/review-banner.tsx, PRD §8.10 / §7.3): renders the review banner and "Destravar para editar" when in review with no foreign lock; when another editor holds the advisory lock it shows "sessão em uso por <name>" and offers no unlock (the lock forces review). State comes from the session store.
- **Non-session route → station resolution** (@/ui/app/App.tsx): `login`/`dashboard`/`setup` resolve the same-named station; an `unknown` top-level path resolves the station named by its first path segment (so `/imports`, whose station lands in ENG-248, renders once its folder exists) and otherwise falls through to the station host's "em construção" placeholder. This is what makes `/setup` reachable now that the router matches it — the dashboard's "Nova sessão" navigates there and lands on the Setup station.
- **Session rehydration + continuous autosave** (@/ui/app/App.tsx `useSessionHydration`, PRD §7.3): on `/session/:id` with an empty in-memory @/ui/state (a browser reload or a resume-from-dashboard, where Setup never populated the live store), it `load`s the persisted DTO from the app-global store and injects `fromSessionDto(dto).state` into `sessionStore`, so the session resumes at its current step instead of stranding on "carregando a sessão…". A `loadedId` ref guards against reloading over a live session (Setup already loaded the freshly-created one) and re-runs when the route id changes. In the same step it **wires the autosave port** (ENG-273): `sessionStore.setAutosave(live => appSessionStore().autosave(routeId, toSessionDto(live, meta)))`, so from then on every domain mutation persists the WHOLE state under this session's `meta` (granularity/audio/consent, captured from the loaded DTO) — without this, only Setup's initial DTO and completion were ever written, and a reload resumed on Escuta 1 losing all decisions. `useAutosaveFlush(routeId)` flushes the adapter's debounced queue on `pagehide` (reload/close) and on session change, closing the debounce window so the last decision is never lost. (Voice-answer `meta.voice` growth across a reload stays a follow-up; the recorder itself is wired below.)
- **Advisory lock on hydration** (@/ui/app/App.tsx `useSessionHydration`, ENG-277, PRD §7.3): after loading the state it also reads `appSessionStore().lockStatus(id)`; if the lock is held by a holder whose `user_id` differs from `DEFAULT_FIXTURE_USER` (@/adapters/sessions), it calls `sessionStore.setLock({ holder: <display_name> })` so opening a session already held by someone else renders the review-only chrome / "sessão em uso por <name>" banner (@/ui/app/review-banner.tsx). Ceiling (ponytail comment): this flow never acquires its OWN lock, so any present lock is treated as foreign — the comparison is against the store's default user.
- **Mid-session auth expiry** (@/ui/app/App.tsx `useAuthExpiry`, ENG-277, PRD §7.1): subscribes the app-global `appAuth().onAuthExpired(() => navigate('/login'))`, so a server token expiring on ANY route — `/session/:id` included — routes back to login WITHOUT touching the in-memory session state (re-login resumes at the same step). Previously this was wired only in the dashboard page, so expiring mid-session did nothing.
- **Station host** (@/ui/app/station-host.tsx): resolves the active station from the registry; a not-yet-built station falls back to a quiet "estação em construção" line, so a station issue enters purely by adding its `ui/pages/<station>/` folder. It accepts optional `stationProps` and forwards them to the resolved component: most stations resolve everything internally (session via `useSessionStore`) and take no props, while stations needing shell-held wiring ports receive them here. Prop-less stations ignore unknown props. @/ui/app/App.tsx passes `{ store, sessionId }` when the Export station is current — `store` is the app-global sessions adapter (@/ui/app/session-adapter.ts), `sessionId` comes from the `/session/:id` route — `{ recorder, player }` when the Mapeamento station is current (the fixture `voice` recorder resolved once from `buildAdapterRegistry().voice.fixture()`, ENG-273, so the interview mic actually records §8.7, plus the audio `Player` to play the phrases), and `{ player }` to the other necklace stations (escuta1/escuta2/triagem/segmentacao) so they play beads/edges/scenes (§8.2, ENG-275). The `player` is the `Player | null` from `useSessionPlayer`; stations already handle `null`. The in-session body lives in a `SessionStations` child that holds the local `viewingExport` flag: clicking the reachable Guardar bead sets it true (landing on Export); navigating any other station clears it and dispatches `setMode` as before. That child is remounted via `key={sessionId}`, so switching sessions resets `viewingExport` — without the remount the flag would leak and a session that never reached the gate would open on Export (the reset avoids `set-state-in-effect`/`set-state-in-render`, both lint errors here).
- **Header** (@/ui/app/header.tsx): Shemá brand (icon + eyebrow "Arquivo Oral · Tripod" + "Colar de Sons" title + subtitle) and the sound toggle (`aria-pressed`, drives `appStore.muted`). Presentational — station audio consults the muted flag; this is v2-only chrome with no golden-harness constraint.
- **Addons layer** (@/ui/app/addons-layer.tsx): a fixed bottom-right overlay region above the station, `pointer-events: none` except on the addons themselves — discreet chrome that never blocks the station. Its first occupant is the tutorial addon (@/ui/app/addons/tutorial.tsx): it reads the session from @/ui/state's `useSessionStore`, derives the current station via @/ui/app/stepper-model.ts (`stepperStations`, the entry whose state is `current`), and renders the @/ui/organisms/tutorial-popup organism keyed by that station; no session (login/dashboard) renders nothing. The popup deliberately renders inline (no Radix portal) so it stays inside `.cds-addons-layer` — never inside the station's `main`.

### Things to Know

- **The shell is edited only for shell-held wiring, and rarely.** The add-a-file registries mean stations/adapters/addons normally land without touching this folder; the durable-shell promise is about registration, not immutability. The exception is wiring a page that cannot resolve everything internally: ENG-270 made the Export tail reachable and injects `{ store, sessionId }` into it, since a completion/download page needs the shell-held sessions store and the route's session id. The registries themselves stay static top-level `import.meta.glob` literals (a Vite requirement) wrapped in factories whose modules-map parameter defaults to the glob, so tests inject a fake map without touching disk; zero matches yields `{}`.
- **The app-global sessions store** (@/ui/app/session-adapter.ts, `appSessionStore()`, ENG-272): ONE lazily-memoized `FixtureSessionStore` singleton over a shared `FixtureSessionBackend`, and the single source of session state for the whole facilitator flow — @/ui/pages/setup, @/ui/pages/dashboard and @/ui/pages/export all resolve it (their `ports.ts` `defaultSessionStore()` returns it), so a session created in Setup appears in the Dashboard list and is readable by Export. The backend persists to a `KeyValueStorage` (the browser `localStorage`, resolved defensively — absent in Web-Storage-less contexts) so sessions survive a reload and feed `useSessionHydration`. Real-mode env selection remains ENG-247.
- **The addons glob eats every direct-child `.tsx`**: `/ui/app/addons/*.tsx` eagerly imports ANY sibling `.tsx` as an addon component. Never place a non-addon `.tsx` directly in `ui/app/addons/` — addon tests live one level down in `ui/app/addons/__tests__/`, outside the one-level glob. The header comment in @/ui/app/addons-layer.tsx still says the directory starts empty in production; that predates the first addon — @/ui/app/registries.test.ts now asserts real glob discovery (the tutorial addon is found) instead of emptiness.
- The player container invariant is load-bearing: never swap the portal container — move the persistent holder node instead, or the whole player subtree remounts (react#12247) and playback dies on every station change.
- **The session player is fixture-only** (@/ui/app/audio-player.ts, ENG-275): the fixture bucket serves audio as `PcmSpec` JSON, not WAV, which only @/adapters/audio `FixtureAudioEngine` decodes — so the real `WebAudioEngine` is deliberately NOT selected here (the env-based real toggle is ENG-247). The clock is driven by the rAF→`advance` bridge (`startClockBridge`), the alternative the issue offered, rather than by a real engine's own rAF.
- Navigation never bypasses the gates: the stepper only calls `onNavigate` for `reachable` stations, and `App` funnels the `setMode` through `sessionStore.apply`, which is a no-op unless `canEdit()` (see @/ui/state/docs.md). Offline/review/lock therefore freeze station changes without losing state.
- Online state has two sinks on purpose: the local flag renders @/ui/organisms/connection-gate immediately, and `sessionStore.setOnline` pauses mutations — both fed from the `connectivity` port subscription AND from window `online`/`offline` events (`useOnline` adds/removes both, cleaning up in the effect teardown). The window bridge is what lets Playwright's `context.setOffline` drive the gate + pause with no app/fixture change (ENG-277) — the "small dev/test connectivity hook" kept entirely in the composition root.
- **The app-global auth singleton** (@/ui/app/auth-adapter.ts, `appAuth()`, ENG-277): ONE lazily-memoized `FixtureAuthProvider` shared by Login, Dashboard (@/ui/pages/dashboard/ports.ts `defaultAuth()` delegates to it) and the session root's `useAuthExpiry`. Because all three surfaces hold the SAME instance, a single expiry trigger (or the login's own token) is observed everywhere — expiring mid-Triagem has someone to notify. Real-mode env selection remains ENG-247.
- **The shared fixture backend is exposed** (@/ui/app/session-adapter.ts, `appSessionBackend()`, ENG-277): the app-global `appSessionStore()` is now built over this lazily-memoized `FixtureSessionBackend` singleton, exported so a test can build a SECOND `FixtureSessionStore` over the SAME backend as a different user — the only way to seed a foreign advisory lock (two Playwright browser contexts have separate localStorage otherwise).
- **Dev-only resilience test seam** (@/ui/app/test-seam.ts + @/ui/app/main.tsx, ENG-277 → acceptance-6 ENG-257): `installTestSeam()` attaches `window.__cds` with `expireAuth()` (→ `appAuth().simulateExpiry()`) and `seedForeignLock(id, holder?)` (acquires the lock via a second store over `appSessionBackend()` as user "Ana"). These are the two resilience triggers with no natural UI/localStorage path; offline needs no seam (`context.setOffline`). @/ui/app/main.tsx imports and installs it lazily ONLY under `import.meta.env.DEV`, so it never ships in the production bundle — the E2E webServer runs Vite dev, where DEV is true.
