# Noridoc: report

Path: @/ui/pages/report

### Overview

- The report station (ENG-250): the consolidated, **editable** mapping report (PRD v2 §8.7 "The report", redesign §6.6). One card per question, in the exact order the domain produces (`questionSequence`), each numbered `Q<n>` by its position in the conversation. **Downloading is the NEXT screen's job** — the footer's single action is "Guardar os documentos →".
- A wiring component: it reads the pure @/domain session through the @/ui/state session store, renders an editable card per question, dispatches text writes through the domain answer store, plays voice answers through the injected `VoiceRecorder` port, and — since ENG-327 — hosts the **review of machine drafts** (transcription + PT→EN translation) coming from the `Transcriber` port (@/adapters/stt).
- **This is where a recorded answer becomes text.** The recording never reaches the artifact; the confirmed English does. That makes this station a required stop, not a nicety: @/contracts `reportExportStatus` blocks the export while any recorded answer still has no confirmed text.
- Facilitator surface (§7.2): unlike the listener screens, digits/IDs are allowed here — the report lists scenes and phrases with numbers, kinds and section headers.

### How it fits into the larger codebase

- **The station after the conversation.** It self-registers through the `import.meta.glob('/ui/pages/*/index.tsx')` registry, and is ALSO resolved directly by @/ui/pages/conversation (a static `import.meta.glob('/ui/pages/report/index.tsx')`) so the "próxima" at the last conversation question opens the report. Both consume the default export.
- **Question order is the domain's.** The flat card order comes verbatim from `questionSequence` (@/domain/mapping.ts): L1 `L1_Q` → per `lockedParts` (none_fit included) `L2_Q` → per `productiveFrases` `L3_Q`. The UI only inserts section (level) and scene/phrase group headers as it walks the sequence; those headers and the card's question text follow the UI language (`questionTextFor`, @/ui/i18n/conversation-questions.ts), while the question's `field` hint still renders the raw domain value.
- **The screen translates, the `.md` does not** (ENG-279). The card text comes from `questionTextFor`, but `buildMapReport` (called only by @/ui/pages/export) reads the PT-BR scripts straight from @/domain — @/contracts cannot import `ui/` at all (a dependency-cruiser rule) — so the exported **questions** are byte-identical regardless of the UI language, and the golden harness keeps that output pinned. The **answers** are a separate matter: the confirmed text is English by product decision (ENG-326), and it is written into `SessionState` by a human action here, never by `t()`.
- **The transcription job is asked for from here.** @/ui/pages/conversation forwards the `Transcriber` port and the session id; the hook @/ui/pages/report/use-stt-drafts.ts starts the job on open and polls until it finishes. The station consumes drafts and never lets one become an answer on its own.
- **Answers.** Text edits write through `setAnswer` (@/domain/mapping.ts) — the lazy answer store created/extended by `ensureMapping`; the store never loses answers. Voice answers are detected through the injected `VoiceRecorder.has(path)` (@/adapters/voice), keyed by the canonical `respostas/level{1,2,3}/…/<k>.webm` path from `voiceAnswerPath` (§10.4/O5); the play affordance calls `recorder.play(path)`.
- Reads session via `useSessionStore`, writes only through `sessionStore.apply` (see @/ui/state/docs.md), so the store's editability gates (online/review/lock) can silently pause a write. Per @/.dependency-cruiser.cjs, @/ui/pages may import @/domain, @/contracts, @/ui/state, @/ui/atoms and adapters.

### Core Implementation

- **Header + white cards** (realigned to the Protótipo): a new `<header>` (eyebrow + serif italic headline, `report.eyebrow`/`report.headline`) sits above the question list; each `ReportCard` is now a white card (`--cds-radius-card-sm`, `--cds-shadow-card`) instead of the previous cream-tinted bordered card. Purely visual — the card content/order is unchanged.
- **`Report({ recorder, preloaded, stt, sessionId })`** (@/ui/pages/report/index.tsx): subscribes `session`; renders `null` until a session with a non-empty question sequence exists. A `useMemo` derives `mapped` = `session` if it already has a mapping, else `ensureMapping(session)`, so answers render before the mount effect persists the store (which calls `sessionStore.apply(ensureMapping)` once when `session.mapping` is null). `stt`/`sessionId` are optional — without either, the station degrades to typing only.
- **Voice discovery, per answer and without a barrier** (ENG-319/ENG-337). One effect probes `recorder.has(path)` per question path and, when there is a recording, reads `recorder.duration(path)` after; it keeps a set of voiced paths, a set of already-CHECKED paths, and a `path → seconds` map. A path the caller already resolved (`preloaded`) is never re-asked. Until a path is checked the card shows a dimmed "procurando" row rather than the empty state — loading is not the same as no answer. The voiced set drives the voice row, decides which paths go to the transcription job, and counts the answers still awaiting confirmation; it never reaches the `.md` builder, which no longer takes a voice argument at all.
- **Draft lifecycle** (ENG-327): the voiced paths go to `useSttDrafts`; a returned `en` is seeded ONCE into the reserved `en__<k>` key, edited freely in place, and confirmed by writing it into the real answer slot through the same `setAnswer` path a typed answer uses. Seeding is guarded twice — a per-path `seeded` ref, and a check that the reserved key does not already EXIST in the bucket — so a re-mount (returning to the session, a reload) cannot resurrect text a human already deleted. `readAnswer` returns `''` for both "never written" and "emptied", which is why existence, not emptiness, is the test.
- **`ReportCard`** renders one question. A typed answer shows an editable `textarea`; a voice-only answer (recording present, no typed text) shows the voice row (▶ play + decorative waveform + duration slot); an empty answer shows the `textarea` with the "ainda sem resposta gravada" placeholder. `ausencia` questions carry the wordless role marker.
- **The card is quiet by design** (prototype parity). The typed answer is a borderless, one-line field that grows with its text (`field-sizing: content` over `rows={1}`): empty, its italic placeholder reads as the prototype's "ainda sem resposta gravada" line, so a facilitator who will not write sees prose, not a form — 41 fixed 64px boxes read as a questionnaire. The note invite is a `+` text link, not a pill.
- **Reserved answer keys are the persistence trick, used twice.** The facilitator note (`nota__<k>`) and the English draft under review (`en__<k>`) are both stored via `setAnswer` in the SAME bucket as the answer they belong to. Because the mapping buckets are free `record<string,string>`, they round-trip through the autosave DTO (@/contracts/session-state.ts) for free — but `buildMapReport` only ever emits the question keys (`L1_Q`/`L2_Q`/`L3_Q`), so neither can appear in the exported `.md` (§10.4: the frozen skeleton has no note or draft lines). "An unconfirmed draft never enters an artifact" therefore holds **by construction**, not by the builder's author remembering to exclude it.
- **`DraftReview`** renders inside a card only while the answer is voice-recorded AND still has no confirmed text. It shows the job's phase (`running` → "transcrevendo…", `failed` → a message plus "tentar de novo"), and once a draft arrives: a "sugestão — revise" badge, the read-only `source` transcript, an editable English textarea, and the confirm button. Once confirmed the card is an ordinary answered card and the block disappears.
- **The live region is registered empty on mount** and only later carries the summary ("N respostas para revisar"). A `role="status"` created together with its content is not announced by screen readers; it also never moves focus (WCAG 2.2 SC 4.1.3) — the reviewer arrives when they choose. It announces the count only, never the drafts themselves.

### Things to Know

- **Voice duration is real** (ENG-271). The voice row's duration slot (`aria-label="duração da resposta"`) shows the saved recording's length formatted `m:ss`, read from `VoiceRecorder.duration(path)` (@/adapters/voice/types.ts). It falls back to `—` only before the durations resolve or when no recorder is wired.
- **The recorder arrives from the conversation.** @/ui/pages/conversation mounts this station with its own `recorder`; without it every card falls back to "ainda sem resposta gravada" and — since the interview is voice-only — no answer would be audible here at all. The `Transcriber` and the session id travel the same way.
- **The polling loop is `setTimeout`, not `setInterval`** (@/ui/pages/report/use-stt-drafts.ts): with an interval a slow poll does not hold the next one and they pile up. It backs off from 2s to a 10s ceiling, gives up after five minutes into a `failed` phase that offers a retry, pauses entirely while `document.hidden` (the browser throttles hidden timers to ~1/min anyway) and polls immediately on return — that is what makes "leave and come back" feel instant.
- **A late response must not win.** A job started before a re-record can answer after it, so every request carries a stamp (`requestId` ref plus a `paths#attempt` key) and a result whose key does not match the current request reads as "still running" and is discarded. Without it a stale answer would silently overwrite newer state. The phase is derived at render, never written from an effect (the house `react-hooks` rule).
- **Manual typing is always available**, at every phase. If the job fails, stalls, or the port is absent, the card's own answer field still resolves the question — §8.7 requires no dead end, and the machine has no authority anyway.
- **No browser test.** This station has no `Necklace`/canvas geometry, so all tests run in jsdom (`report.test.tsx`): rendering per answer type + role marker, text edits reaching the domain answer store, the facilitator note persisting across a re-mount while staying out of the `.md` (byte-asserted against the real builder), and the draft flow — running/arrived/confirmed, editing before confirming, no draft for an unrecorded question, and the pre-registered live region. The hook has its own suite (`use-stt-drafts.test.ts`). There are **no export actions on this station**: completing and downloading live in @/ui/pages/export, and its tests own the export gate.
