# Noridoc: adapters/sessions

Path: @/adapters/sessions

### Overview

- The session-persistence port (ENG-240, PRD v2 §7.3): continuous full-state autosave (no explicit save action), resume from any machine, the completed/in-progress lifecycle with the opaque artifact trio, a single-editor advisory lock, and the `respostas/*.webm` voice resources (put/get/list/delete).
- `SessionStore` (@/adapters/sessions/types.ts) is the port; it ships the headless `FixtureSessionStore` (default) and the real `HttpSessionStore` — the tripod-api client whose wire was verified end-to-end in ENG-267 (`http/sound-necklace.http` on the API side) — plus the reusable `createAutosaver` state machine shared by both, following the same port + real + fixture + `register.ts` shape as @/adapters/audio and @/adapters/connectivity (see @/adapters/docs.md).
- Exists so the whole app, dashboard, and every UI/E2E test run with no real backend; the real mode is selected by the composition root under `VITE_API_MODE=real` (@/ui/app/session-adapter.ts, ENG-247), fixture stays the default.

### How it fits into the larger codebase

- Consumes @/contracts DTOs (`SessionStateDto`, `SessionSummary`, `ArtifactTriple`, `LockStatus`, `SessionStep`, `ResourcePath`, `CreateSessionRequest` from @/contracts/api.ts and @/contracts/session-state.ts). It imports the connectivity port (@/adapters/connectivity/types.ts) for the offline pause; it needs nothing from @/domain and never imports `ui/` — @/.dependency-cruiser.cjs enforces both directions.
- @/adapters/sessions/register.ts still declares the `{ port: 'sessions', fixture, real }` registration per convention (@/docs/architecture.md §4), but the app's actual wiring is the mode-aware app-global singleton `appSessionStore()` in @/ui/app/session-adapter.ts: under `VITE_API_MODE=real` it builds the `HttpSessionStore` with the browser fetch/connectivity, the logged-in editor as a `user` thunk, an **async** token gated behind `authReady()` (every session call waits for the boot resume, so a reload cannot race a 401), and `onLockLost` feeding the review chrome.
- Consumed by `ui/` only through the port and props/wiring (Setup creates, Dashboard lists/resumes, Export completes/reads artifacts). The listener-facing autosave freeze pairs with the connectivity gate (@/adapters/connectivity/docs.md) — losing connection pauses saving without clearing state.

```
Setup ─create()─▶ SessionSummary ──▶ Dashboard list()/get()
  │ autosave(state) (debounced, offline-paused)
  ▼
FixtureSessionStore ── over ──▶ FixtureSessionBackend (shared "server")
  │  clone in / clone out (opaque custody)      │ optional mirror
  │                                             ▼
Export ─complete(state, artifacts)─▶      KeyValueStorage (localStorage)
        getArtifacts() byte-identical      key colar-de-sons:sessions:v1
```

### Core Implementation

- **`createAutosaver`** (@/adapters/sessions/autosave.ts) — the shared autosave machine both stores delegate to. It coalesces per session via a `Map` (last state wins), debounces bursts with `setTimeout`, retries a failed `persist` with exponential backoff, and **pauses while offline**: `drain`/`persistOne` early-return when the injected `ConnectivityMonitor` reports offline, keeping the pending state; a `subscribe` callback re-arms the timer on reconnect so nothing is lost (PRD §13). One failure is never retried: a thrown `LockLostError` (below) is terminal — the loop drops the pending state instead of holding it for a retry that would only repeat the same rejection. The persistence target is injected, so the fixture writes to its backend and the HTTP store issues a PUT. It also exposes an aggregate `onStatus(cb)` (surfaced on the port as `SessionStore.onAutosaveStatus`) that reports `'saving'` while anything is pending or in flight — including offline and retry, so the UI never claims "saved" with a write still open — and `'saved'` once everything has landed; the app's header binds this to the save-chip (ENG-351).
- **`FixtureSessionStore`** (@/adapters/sessions/fixture.ts, the DEFAULT) — built over a **shareable** `FixtureSessionBackend` (the simulated server). Two stores over one backend are two users on one server, which is what exercises the advisory lock. The backend optionally mirrors its JSON-able records (summaries/state/artifacts/lock) to a `KeyValueStorage` (localStorage shape) under `colar-de-sons:sessions:v1` and hydrates from it on construction; voice-resource bytes live in memory only. The dashboard step (`SessionSummary.progress.current_step`) is derived from the saved state's `mode`/status via `stepFor`.
- **`HttpSessionStore`** (@/adapters/sessions/http.ts) — the real tripod-api client, all routes under the `/sound-necklace` prefix, with an **injected** `fetch` (no network in CI). The token getter may return a Promise (the real wiring awaits `authReady()` inside it) and `user` may be a thunk (the editor is only known after login — `me` is a getter that evaluates it). JSON responses that are contract-shaped (`SessionSummary`, list, `LockStatus`) are schema-parsed; the opaque state and the WebM resource bytes travel without re-serialization. `acquireLock` and `renewLock` both `PUT /sessions/{id}/lock` — the backend serves acquire and renew as the same idempotent operation, and there is no POST on that route (it would 405).
- **`complete()` is a three-step wire** (§8.8/§10.5): the session's pending autosave is cancelled **and the in-flight write is awaited** (`Autosaver.settle` — a dispatched `PUT /state` cannot be aborted, and landing after the final state would regress the saved document; `reopen` does the same), then a final `PUT` of the state, then `POST /sessions/{id}/artifacts` as **multipart form-data carrying the raw artifact bytes** — field names `manifest`/`anchoring`/`report`, filenames frozen from PRD §10 (`bead-manifest.json`, `anchoring-return.json`, `mapping-report.md`), no hand-set content-type so the runtime signs the multipart boundary — validated against the upload receipt (`ArtifactUploadResponseSchema`, per-artifact checksums), then a **bodyless** `POST /complete`. The API never re-serializes the bytes; byte-identity on the way back was proven against the real bucket in ENG-267.
- **Downloads ride signed URLs.** `getArtifacts` issues one GET per kind; the route answers 307 to a signed URL and `fetch` follows it alone — the browser strips the `Authorization` header on the cross-origin hop (fetch spec), which the signed URL requires. `getResource` is an explicit two-hop: `GET /resources/url?path=…` (with Bearer) returns the signed URL, then a second fetch **without** Bearer (an `Authorization` header alongside the signature is a 400 on GCS). Resource writes/deletes address the path via the `?path=` query (`putResource`/`deleteResource`); `listResources` lists the whole session and filters the prefix client-side.
- @/adapters/sessions/index.ts is the public barrel for the wiring layer and tests; only the composition root imports the implementations (to build the app-global singleton in @/ui/app/session-adapter.ts) — stations resolve the store through their `ports.ts` defaults, never by importing a class.

### Things to Know

- **Custody is opaque (§10.5):** state and artifacts round-trip byte-identical. The fixture deep-clones (`structuredClone`) both in and out and never re-serializes or reinterprets the payload shape; `HttpSessionStore.load`/`getArtifacts` return the raw payload as-is (deep validation is the state layer's job, not the store's). This byte-identity is what keeps the golden harness green.
- **Acquiring or reading the lock never throws on conflict.** `acquireLock`/`renewLock`/`lockStatus` all return a `LockStatus`; if `holder` belongs to someone else, the caller opens review-mode (§7.3). Every `SessionStore` exposes its own editor identity as `me` so a caller can compare `holder.user_id` against it without knowing the mode (fixture/HTTP) or importing a fixture constant. The fixture's lock is per-user with a TTL (`lockTtlMs`, default 30s) and self-expires if the holder disappears — `#heldByOther`/`lockStatusOf` treat an expired lock as free; the real backend's lease (ENG-262) is a fixed 60s, renewed by the same `PUT` used to acquire.
- **Two different 409s live on the state PUT, and the body's `code` separates them** (fencing, ENG-262). `HttpSessionStore.#send` parses the error body into `ApiError.body`; the autosave `persist` treats only `code: 'SESSION_LOCKED'` as the verdict — it reports once via `onLockLost(id, holder)` (autosave is fire-and-forget, so the callback is the only way the UI learns; `holder` is the body's `holder_name`, null if absent) and rethrows `LockLostError` (which carries the same `holder`), terminal for `createAutosaver`: no retry, pending state dropped. `SESSION_LOCK_CHANGED` (the lease lapsed mid-write) and any 409 without a known code stay **transient** — retried with backoff exactly like a 5xx. `complete`/`reopen` skip the translation: a 409 there reaches the caller as the raw `ApiError` (@/adapters/api). The fixture never fences writes at all, so this path only exercises against `HttpSessionStore`.
- `load`/`getArtifacts` on a session that was never saved/completed throw `SessionNotFoundError` (also raised for an unknown id), the typed surface the UI shows copy for.
- Both stores' `autosave` schedule a **clone** of the state (`structuredClone`) so a later caller mutation cannot rewrite an already-queued save; the coalescing map still lets a genuinely newer `autosave` win.
- `flush` and the autosaver are **no-ops while offline** — a forced flush on navigation only persists when online; the pending state waits for reconnect.
- The fixture's `latencyMs`/`#settle` inject optional per-op delay for realistic wiring tests; tests otherwise drive the autosaver with fake timers (debounce/backoff are `setTimeout`-based).
- Fixture-safe adapter PRs may merge autonomously on green (unlike @/domain and @/contracts).
